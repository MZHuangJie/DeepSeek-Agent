# Code Review 模式

你正在执行 /code-review 命令，需要对代码进行全面审查。

## 审查维度

### 1. 安全性（CRITICAL）
- 命令注入风险：exec/execSync 的参数是否来自用户输入
- 路径穿越：文件操作路径是否经过安全校验
- 密钥泄露：是否硬编码了 API Key、Token、密码
- SQL 注入：数据库查询是否使用了参数化查询
- XSS：用户输入是否未经转义直接渲染

### 2. 代码质量（HIGH）
- 函数是否过长（>50行）
- 文件是否过大（>800行）
- 嵌套是否过深（>4层）
- 命名是否清晰、有意义
- 错误处理是否完备（是否吞掉了异常）

### 3. 可维护性（MEDIUM）
- 是否有重复代码可以抽取
- 模块耦合度是否合理
- 是否有未使用的导入、变量、函数
- 接口设计是否清晰

### 4. 性能（MEDIUM）
- 是否有 N+1 查询
- 是否有不必要的同步操作可以改为异步
- 是否有可以缓存的计算结果

## 输出格式

```
## 代码审查报告

### CRITICAL（必须修复）
- 🔴 问题描述 | 文件:行号 | 修复建议

### HIGH（应该修复）
- 🟡 问题描述 | 文件:行号 | 修复建议

### MEDIUM（考虑修复）
- 🟠 问题描述 | 文件:行号 | 建议

### LOW（可选优化）
- ⚪ 问题描述 | 文件:行号 | 建议

### 总结
- 审查文件数：N
- 发现问题：X 个 CRITICAL + Y 个 HIGH + Z 个 MEDIUM
- 整体评价：...
```

## 执行步骤
1. 先用 `list_files` 了解待审查的文件范围
2. 用 `read_file` 逐文件读取
3. 用 `grep` 搜索常见问题模式（如 `exec(`, `innerHTML`, `dangerouslySetInnerHTML`）
4. 按严重级别汇总输出
