# Plan 模式

你正在执行 /plan 命令，需要为当前任务制定一个完整的实施计划。

## 核心流程

### 第一阶段：需求理解
1. 理解用户的任务目标，如果需求不清晰，**先向用户提问澄清**
2. 用自己的话复述需求，确认理解正确后再继续

### 第二阶段：技术选型（必须做！）
当涉及框架、工具、架构等决策时，必须列出可选方案：
- 框架：React / Vue / Svelte / Vanilla JS
- 样式：CSS Modules / Tailwind / styled-components / 纯 CSS
- 状态管理：Zustand / Redux / Context / 无
- 构建工具：Vite / Webpack / 无构建
- 数据存储：内存 / localStorage / IndexedDB

**如果有多个合理选项，必须调用 `present_choices` 工具弹出选择框**。
不要在消息里只列文字，要用工具弹出的交互式选择框。

### 第三阶段：代码库分析
1. 使用 `list_files` 了解项目目录结构
2. 使用 `grep` 搜索相关代码模式和关键函数
3. 使用 `read_file` 读取关键文件的完整内容
4. 识别现有架构模式、代码约定、工具函数

### 第四阶段：架构设计
1. 设计模块划分、数据流、组件树
2. 设计目录结构
3. 定义关键接口和类型

### 第五阶段：实施计划
1. 将任务拆分为**具体、可验证**的子任务
2. 每个子任务说明：
   - 要修改哪些文件
   - 具体改动内容
   - 预期结果
   - 依赖关系
3. 标注风险点和需要特别注意的地方

## 输出格式

```
## 需求理解
（复述需求）

## 项目现状分析
（关键文件、现有模式、限制条件）

## 方案对比
| 方案 | 优点 | 缺点 | 复杂度 | 推荐 |
|------|------|------|--------|------|
| A    | ...  | ...  | 低/中/高 | ✓   |
| B    | ...  | ...  | 低/中/高 |      |

## 推荐方案
（详细说明选中的方案）

## 实施步骤
1. [ ] 步骤1 | 文件：xxx | 核心代码：`...` | 预期结果（无依赖）
2. [ ] 步骤2 | 文件：xxx | 核心代码：`...` | 预期结果（依赖步骤1）

-> 最终用 write_file 保存为 .{项目文件夹名}-plans/<任务名>.md

## 风险与注意事项
- ⚠️ 风险1：xxx
- 💡 建议：xxx
```

## 铁律
- **必须先做技术选型**，无论新项目还是改现有项目
- **技术选型有多个选项时必须用 present_choices 工具**，不要只用文字列表
- 分析代码库之前不要提方案
- 计划要具体到文件级别，不要泛泛而谈
- **计划完成后用 write_file 保存为 `.{项目文件夹名}-plans/<任务名>.md`**，包含每个步骤的涉及文件、核心代码
- **后续执行时严格按此文档内容实施，不得偏离**
