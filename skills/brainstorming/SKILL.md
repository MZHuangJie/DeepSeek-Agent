# Brainstorming 模式

你正在执行 /brainstorming 命令，需要对用户的问题进行深入分析和创意探索。

## 核心原则

- **先理解，后建议**：在提出任何建议之前，确保完全理解用户的问题
- **广度优先，再深度**：先列出所有可能的方向，再深入分析最可行的
- **结构化输出**：用清晰的结构整理思路，避免混乱

## 执行流程

### 1. 问题拆解
- 用户的核心目标是什么？
- 有哪些约束条件（技术栈、时间、性能要求）？
- 有哪些隐含需求？

### 2. 背景调研
- 使用 `grep` 和 `read_file` 了解项目现有实现
- 搜索相关技术的最佳实践
- 了解类似问题的解决方案

### 3. 方案头脑风暴
- 列出 3-5 个可能的解决方向
- 每个方向说明核心思路和适用场景
- 不做过早的判断和筛选

### 4. 深度分析
- 选择最可行的 2-3 个方向深入分析
- 评估技术可行性、工作量、风险
- 给出推荐和理由

## 输出格式

```
## 问题分析
（对用户问题的拆解和理解）

## 背景信息
（项目相关代码、技术约束等）

## 可选方向

### 方向 A：...
- 核心思路：
- 适用场景：
- 潜在风险：

### 方向 B：...
（同上）

## 深度分析
（对推荐方向的详细分析）

## 建议
（最终推荐和下一步行动）
```

## 注意事项
- 不要在未充分理解问题的情况下就给出答案
- 保持开放心态，不要过早排除"不寻常"的方案
- 如果信息不足，主动使用工具收集信息
